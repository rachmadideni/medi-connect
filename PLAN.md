## Phase 1 — Database Schema

> **Database:** AlloyDB for PostgreSQL
>
> AlloyDB's `google_ml_integration` extension exposes an `embedding()` SQL function that calls Vertex AI **directly from within SQL** — no application-side embedding code required. Embeddings are generated at `INSERT` time and at search time inside the query itself.

### Step 1 — Enable extensions

Run in **AlloyDB Studio** (SQL editor in the Google Cloud Console):

```sql
CREATE EXTENSION IF NOT EXISTS google_ml_integration CASCADE;
CREATE EXTENSION IF NOT EXISTS vector;
```

> ⚠️ Wait for your AlloyDB instance to finish provisioning before running these.

---

### Step 2 — Grant Vertex AI User role to the AlloyDB service account

Run from **Cloud Shell Terminal**:

```bash
PROJECT_ID=$(gcloud config get-value project)
gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member="serviceAccount:service-$(gcloud projects describe $PROJECT_ID \
    --format='value(projectNumber)')@gcp-sa-alloydb.iam.gserviceaccount.com" \
  --role="roles/aiplatform.user"
```

> ⚠️ IAM changes can take a few minutes to propagate before the `embedding()` function works.

### Table: `doctors`

```sql
CREATE TABLE doctors (
  id               SERIAL          PRIMARY KEY,
  name             VARCHAR(255)    NOT NULL,
  specialty        VARCHAR(100)    NOT NULL,
  location         VARCHAR(255)    NOT NULL,
  rating           NUMERIC(3, 2)   NOT NULL DEFAULT 0,
  reviews          INTEGER         NOT NULL DEFAULT 0,
  available        VARCHAR(50)     NOT NULL,        -- "Today", "Tomorrow", "Wed", etc.
  experience       INTEGER         NOT NULL DEFAULT 0,
  avatar           VARCHAR(10),                     -- initials fallback, e.g. "SM"
  image_url        TEXT,                            -- GCS public URL
  bio              TEXT,
  languages        TEXT[]          NOT NULL DEFAULT '{}',
  education        VARCHAR(255),
  created_at       TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

  -- ★ VECTOR COLUMN
  -- 768-dimensional embedding generated by Gemini text-embedding-005
  -- Source text: name || ' ' || specialty || ' ' || location || ' ' || bio
  -- Populated at INSERT time via AlloyDB's built-in embedding() SQL function
  -- (google_ml_integration extension — no application-side embedding code needed)
  -- Used for AI natural language search via cosine similarity (<=> operator)
  search_embedding VECTOR(768)
);

-- HNSW index for fast approximate nearest-neighbour search (cosine distance)
CREATE INDEX idx_doctors_embedding
  ON doctors USING hnsw (search_embedding vector_cosine_ops);

-- Indexes for filter queries
CREATE INDEX idx_doctors_specialty  ON doctors(specialty);
CREATE INDEX idx_doctors_available  ON doctors(available);
```

### Step 4 — Seed doctor data with in-database embeddings

The `embedding()` function is called **inside the INSERT statement** — AlloyDB calls Vertex AI directly, no round-trip through the application:

```sql
-- Insert a doctor and generate the search embedding in one statement
INSERT INTO doctors (
  name, specialty, location, rating, reviews, available,
  experience, avatar, image_url, bio, languages, education,
  search_embedding
) VALUES (
  'Dr. Sarah Mitchell',
  'Cardiologist',
  'New York, NY',
  4.9, 214, 'Today', 14, 'SM',
  'https://storage.googleapis.com/<BUCKET>/dr-sarah-mitchell.png',
  'Board-certified cardiologist with over 14 years of experience in interventional cardiology.',
  ARRAY['English', 'Spanish'],
  'Harvard Medical School',
  -- ★ embedding() calls Vertex AI text-embedding-005 and returns VECTOR(768)
  embedding('text-embedding-005',
    'Dr. Sarah Mitchell. Cardiologist at New York, NY. ' ||
    'Board-certified cardiologist with over 14 years of experience in interventional cardiology. ' ||
    'Languages: English, Spanish'
  )::vector
);
```

Repeat for all 10 doctors. To backfill an existing row:

```sql
UPDATE doctors
SET search_embedding = embedding(
  'text-embedding-005',
  name || '. ' || specialty || ' at ' || location || '. ' || COALESCE(bio, '') ||
  '. Languages: ' || array_to_string(languages, ', ')
)::vector
WHERE search_embedding IS NULL;
```

> **Note:** `VECTOR(768)` is the correct dimension for `text-embedding-005`. Do not mix models — a different model may produce a different number of dimensions and break all inserts.

## Phase 2 — Backend API

### 2a. New project structure

```
backend/src/
  routes/
    doctors.ts        # GET /api/doctors, GET /api/doctors/:id
    search.ts         # GET /api/doctors/search?q=...  (AI vector search)

  services/
    auth.ts           # Verifies Google ID tokens using google-auth-library
    # NOTE: No embedding.ts needed — embeddings are generated inside SQL
    #       using AlloyDB's embedding() function (google_ml_integration)
  middleware/
    requireAuth.ts    # Express middleware — verifies Bearer token, attaches req.user
```

### 2b. REST Endpoints

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `GET` | `/health` | — | DB connectivity check |
| `GET` | `/api/doctors` | — | List doctors (supports `?specialty=&available=`) |
| `GET` | `/api/doctors/:id` | — | Get single doctor with slots |
| `GET` | `/api/doctors/search?q=` | — | AI vector search via cosine similarity |
| `POST` | `/api/auth/verify` | — | Verify Google ID token, upsert user, return profile |
| `GET` | `/api/appointments` | ✅ Required | List authenticated user's appointments |
| `POST` | `/api/appointments` | ✅ Required | Book an appointment |
| `PATCH` | `/api/appointments/:id/cancel` | ✅ Required | Cancel an appointment |

### 2c. AI Vector Search — `/api/doctors/search`

**Flow:**

1. Receive `q` (plain language query string) from the frontend
2. Pass `q` directly as a SQL parameter — AlloyDB calls `embedding('text-embedding-005', q)` **inside the query** to embed the search term on the fly
3. Run cosine similarity against `doctors.search_embedding`:

```sql
-- $1 is the raw query string — AlloyDB calls Vertex AI inline
SELECT
  d.*,
  1 - (d.search_embedding <=> embedding('text-embedding-005', $1)::vector) AS similarity
FROM doctors d
WHERE d.search_embedding IS NOT NULL
ORDER BY d.search_embedding <=> embedding('text-embedding-005', $1)::vector
LIMIT 10;
```

4. Join with `doctor_slots` and return ranked results as JSON

> **Key advantage:** The application sends a plain string (`$1`) — no embedding SDK call in Node.js. AlloyDB handles the Vertex AI round-trip internally.

### 2d. No application-side embedding service needed

Unlike a typical vector search setup, **this project does not need a `services/embedding.ts`**.

AlloyDB's `google_ml_integration` extension provides the `embedding()` SQL function that:
- Calls `text-embedding-005` on Vertex AI from within SQL
- Works in both `INSERT` (seed time) and `SELECT` (search time) statements
- Requires only the "Vertex AI User" IAM role on the AlloyDB service account (see Phase 1, Step 2)

### 2e. `requireAuth` middleware

```ts
// middleware/requireAuth.ts
// 1. Read Authorization: Bearer <google_id_token> from request header
// 2. Verify token with google-auth-library OAuth2Client.verifyIdToken()
// 3. Look up user in users table by provider_id
// 4. Attach req.user = { id, email, name } — reject with 401 if missing
```

## Phase 4 — Auth: Replace In-Memory State

### 4a. Provider

**Google Identity Platform** (already on GCP — no extra cost for basic OAuth).

- Frontend: `@react-oauth/google` package
- Backend: `google-auth-library` package

### 4b. Frontend changes

**`AuthContext.tsx`** — replace `useState(false)` with:

```ts
type User = { id: string; email: string; name: string };
type AuthContextType = {
  isAuthenticated: boolean;
  user: User | null;
  token: string | null;
  signIn: () => void;   // triggers Google OAuth popup
  signOut: () => void;
};
```

Flow:
1. `signIn()` → Google OAuth popup → receive `id_token`
2. `POST /api/auth/verify` with `{ id_token }` → backend upserts user, returns `{ user, token }`
3. Store in context + `sessionStorage` (survives page refresh)
4. `signOut()` → clear context + sessionStorage + Google session

**`SignInButton.tsx`** — wrap with `<GoogleOAuthProvider clientId={...}>` or keep existing style triggering the updated `signIn()`.

### 4c. Backend `POST /api/auth/verify`

```ts
// routes/auth.ts
// 1. Receive { id_token } from request body
// 2. google-auth-library: client.verifyIdToken({ idToken, audience: CLIENT_ID })
// 3. Extract { sub, email, name } from ticket.getPayload()
// 4. INSERT INTO users ... ON CONFLICT (provider_id) DO UPDATE SET name = ...
// 5. Return { user: { id, email, name } }
```

### Cloud Run deployment

```bash

gcloud beta run deploy medi-connect-backend \
  --source ./backend \
  --region asia-southeast1 \
  --set-env-vars \
    PORT=8080,\
    GEMINI_API_KEY=<KEY>,\
    DATABASE_URL=<ALLOYDB_CONNECTION_STRING>,\
    GCS_BUCKET_NAME=<BUCKET>,\
    GOOGLE_CLIENT_ID=<OAUTH_CLIENT_ID>,\
    CORS_ORIGIN=https://<FRONTEND_RUN_URL>

  --source ./frontend \
  --region asia-southeast1 \
  --set-env-vars VITE_API_URL=https://<BACKEND_RUN_URL>
```

> Backend Cloud Run service needs a **VPC connector** to reach AlloyDB on the private network.
